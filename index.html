Introduction

ScratchX is a web based GUI that offers easy to use programming blocks that make it ideal in a classroom setting for the teaching of programming. Unlike its Scratch counterpart, ScratchX supports experimental extensions, which allows users to load their own extensions into the GUI. These extensions are written in JavaScript and can be loaded into the GUI by inserting a Github hosted JavaScript link (e.g. http://leedsrobotics.github.io/practiseblocks.js) into the home page of the ScratchX site (http://scratchx.org).

Platform Compatibility

Since the ScratchX GUI is web based, it can be accessed via any computer and operating system with an internet connection and a recent version of Adobe Flash installed. However, in order to interface with hardware, a hardware plugin must be downloaded from:

https://scratch.mit.edu/info/ext_download/

This hardware plugin is NOT available on Linux. The official plugins are currently only available for download on Windows and Mac.

Browser Compatibility

Though the plugin is available on multiple browsers, Mozilla Firefox is recommended as it comes with the least issues, especially when it comes to getting the hardware plugin to be detected. Before starting the extensions, you should disable the cache beforehand as Firefox repeatedly caches versions of the JavaScript files hosted on the site. This can be done by following these steps:

  • type in the address bar about:config
  • then press the button I'll be careful I promise
  • then type in the bar browser.cache.disk.enable
  • then double click on it, to make it false
  • exit firefox and restart-it.
  • do the same with browser.cache.memory.enable

Alternatively, you can hit (Ctrl + F5) to update the cache with a new version of the page.

Setting up your computer

Before starting the setup process, be aware that interfacing with hardware via ScratchX is only possible on Windows and Mac. This is because of there not being a offical Linux Hardware plugin release for ScratchX that allows hardware to be controlled via JavaScript running on the ScratchX site.

Adobe Flash Player

Since the ScratchX GUI is web based, it can be accessed via any computer and operating system with an internet connection and a recent version of Adobe Flash Player installed. If you are unsure whether your version of Adobe Flash Player is recent enough, quickly visit the ScratchX site (http://scratchx.org/) and open the editor via the link at the bottom of the page. If the editor loads without error, your Adobe Flash Player version is compatible, otherwise you will need to download the latest version of Adobe Flash Player:

https://get.adobe.com/flashplayer/

Github

In order to use JavaScript code on the ScratchX website, you will need a Github page. If you do not already have one, you will need to set one up. If this is the case, follow this simple tutorial that goes through setting up a Github page:

https://pages.github.com

By the end of the tutorial, you should have a Github page that can be accessed via the format myusername.github.io.

In addition, you will also need a crossdomain.xml file that gives Flash permission to access public repositories on your github page. Here is an example that is suitable for use in ScratchX:

<cross-domain-policy xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:noNamespaceSchemaLocation="http://www.adobe.com/xml/schemas/PolicyFile.xsd">
  <site-control permitted-cross-domain-policies="master-only"/>
  <allow-access-from domain="scratchx.org"/>
  <allow-access-from domain="llk.github.io"/>
</cross-domain-policy>

This file should be placed in the root of the host, meaning that it should be accessible via the format myusername.github.io/crossdomain.xml.

Loading JavaScript files into ScratchX

In order to load a JavaScript extension, follow these steps:

  1. Copy the link the JavaScript file hosted on a Github page (e.g. myusername.github.io/practiseblocks.js)
  2. Go to http://scratchx.org/
  3. Under the section labelled 'Open an Extension URL', paste the JavaScript link and click open (your browser may not respond for a few seconds)
  4. The ScratchX editor should now be loaded with your extension. If the editor loaded and any of your blocks are missing from the 'More Blocks' section, open the browser console to check if any errors occurred.
Hardware Interfacing

In order for ScratchX to be able to interface with USB and Serial devices, you must install the hardware plugin offered by ScratchX. This hardware plugin is only available on Windows and Mac, so Linux users can not interface with hardware using the official ScratchX plugin. In addition, this plugin often has issues being detected on certain browsers, so the use of Mozilla Firefox is recommended. The main versions of the plugin are hosted here:

https://scratch.mit.edu/info/ext_download/

If the versions hosted there don't work, there is a Beta version of the plugin here:

Windows: https://www.mediafire.com/?d99e82e7w8k7x8l

Once installed, check that the plugin is enabled in your browser settings. Once its enabled, you can check if its working by loading a hardware extension (e.g. http://khanning.github.io/scratch-arduino-extension/arduino_extension.js ). Once the extension is loaded, there should be a coloured circle next to the extension title in the 'More Blocks' section. By hovering over it, it will tell you the state of the extension. The states include:

Red: Missing Browser Plugin. If you receive this error, try reinstalling the plugin or install the beta version.

Yellow: Plugin is working but no device is detected. Ensure your device connected to the computer.

Green: No errors detected. Extension is ready.

Starting a new extension

Here is the template in which you can create and add new blocks:

(function(ext) {
    // Cleanup function when the extension is unloaded
    ext._shutdown = function() {};

    // Status reporting code
    // Use this to report missing hardware, plugin or unsupported browser
    ext._getStatus = function() {
        return {status: 2, msg: 'Ready'};
    };

    // Block and block menu descriptions
    var descriptor = {
            blocks: [
                    ]
    };

    // Register the extension
    ScratchExtensions.register('Sample extension', descriptor, ext);
})({});

The above template will open in ScratchX without error but won't do anything since the array of blocks inside the descriptor is empty. In order to create a block, we need to insert an array into the block array. The elements of the array will vary in format depending on the type of block you wish to create. Here is a break down of how to declare different types of blocks

Creating Blocks

Block Types
var descriptor = {
    blocks: [
        ['', 'The Text On the Block', 'function_name'],
    ]
};

To create a block, you first need to create an array inside the blocks array. This array must consist of 3 elements. The first is a character that refers to the type of block you wish to create, the second is the text that appears on the block itself (should be used to describe what the block does) and the third is the name of the function that will run when the block is run. In addition, you can add additional elements to the array which will be passed in as parameters to the function. For example, ['', 'The Text On the Block', 'function_name', 2, 3] will pass in 2 and 3 in as parameters to the function function_name in that order.

Here is a list of the available blocks types that you can create by the declaring the corresponding character as the first element in a new blocks array:

'' or ' ' : Synchronous Command Block

'w' : Asynchronous Command Block (useful if you want to wait before finishing)

'r' : Synchronous Reporter Block

'R' : Asynchronous Reporter Block (useful if you want to wait before finishing)

'h' : Synchronous Hat Block (Must return true or false)(If true, runs stack)

Fields

In addition, you can also add fields to blocks so users can pass in values to the function. There are 3 different types of fields:

%s : String Field

%n : Number Field

%m.menuname : Menu Field

Note the .menuname at the end of the menu field. This refers to a optional, second element in that can be added to the descriptor if you wish to use menus. The menu should be declared as a object and contain the property name that corresponds to the name specified in one of your fields (e.g. menuname), followed by an array of values as the property value. These values will be the options of the menu. It should also be noted that if you specify parameters in the array of a block that uses fields, these parameters will fill the fields as default values.

Here is an example that makes use of all the field types:

var descriptor = {
    blocks: [
        ['', 'Say %s , %n times %m.direction', 'function_name', 'hello', '5', 'backwards'],
    ],
    menus: {
        direction: ['forwards', 'backwards']
    }
};

The default appearance of this block will be a command block that says 'Say [hello], [5] times [backwards]'

Note: Blocks are rendered in order of their positions as a element, so if you wish to order your blocks in a certain way, simply move the elements inside the blocks array.

Functions

Declaring Functions

Creating functions for an extension is not much different that creating an ordinary function. The main difference is in how they are declared.

(function(ext) {
    // Cleanup function when the extension is unloaded
    ext._shutdown = function() {};

    // Status reporting code
    // Use this to report missing hardware, plugin or unsupported browser
    ext._getStatus = function() {
        return {status: 2, msg: 'Ready'};
    };

    // Block and block menu descriptions
    var descriptor = {
            blocks: [
                    ]
    };

    // Register the extension
    ScratchExtensions.register('Sample extension', descriptor, ext);
})({});

Looking back at the template, note the parameter ext mentioned at the beginning of the function. This object is used to contain all the functions that are used by the blocks in the descriptor by declaring them as properties or methods of the object. So a an example of a function declaration could be:

ext.myFunction = function() {
    // Insert Code here
}
Built-in Functions

There are several built in functions that run on certain events that you may find useful. Here are few of the commonly used functions:

ext._deviceConnected: This function runs whenever a new device is connected. Often used for creating initial connections.

ext._deviceRemoved: This function runs whenever a device is removed (serial devices may not be detected when removed). Often used for closing connections.

ext._shutdown: This function runs when the extension is closed. Often used to close connections if a device is still plugged in when the extension closes.

ext._getStatus: Required to provide error reporting.

Serial Support

If the examples for serial support on the ScratchX wiki don’t work, here is an example of a barebones serial extension that connects to the first available serial device, printing out data it receives:

(function(ext) 
{
    var device; // Declares undefined device
    var dataView = null; // View to contain received data


    /**
     * Return status of the extension
     */
    ext._getStatus = function() 
    {
        return {status:2, msg: 'Ready'};
    };


    /**
     * Checks if newly connected devices are viable (runs whenever new device is connected)
     */
    var potentialDevices = [];
    ext._deviceConnected = function(dev) {
        potentialDevices.push(dev); // Add new device to the list of potential devices

        // If no device is currently set, try another device
        if (!device) {
            tryNextDevice();
        }
    }


    /**
     * Checks if a compatible device is connected, opening a connection if it is and specifies data received
     * should be printed in a readable format
     */
    function tryNextDevice() 
    {
        device = potentialDevices.shift(); // Set device to the next available device
        if (!device) return; // If there was no device available, return

        device.open({ stopBits: 0, bitRate: 9600, ctsFlowControl: 0 }); // Opens connection

        // When data is received from device, convert the data to a readable format and print to console
        device.set_receive_handler(function(data) {
            dataView = new Uint8Array(data);
            for(var x = 0; x < dataView.length; x++)
            {
                console.log(String.fromCharCode(dataView[x]));  
            }
        });
    };


    /**
     * Processes that run on extension shutdown
     */
    ext._shutdown = function(){};


    // Registers block types, names and corresponding procedures
    var descriptor = {
        blocks: []
    };


    var serial_info = {type: 'serial'}; // Declare type of hardware required

    ScratchExtensions.register('Extension Name', descriptor, ext, serial_info); // Registers the extension
})({});

Note the additional parameter inside the ScratchExtension.register method. This parameter is required if you are using serial devices but is otherwise optional.

Data sent to a serial device in this way must be stored in an ArrayBuffer. However, it must be contained within a TypedArray, such as the Uint8Array for unsigned 8-bit integers seen above. A list of available TypedArrays can be found here:

https://developer.mozilla.org/en-US/docs/Web/JavaScript/Typed_arrays

The data can then be sent by getting the ‘buffer’ attribute of the TypedArray object inside the parameters of the send method of the device object (e.g. device.send(yourTypedArray.buffer); ). Here is an example of a function that sends the command “@id” to a connected device:

ext.idRequest = function()
{
  var commandLeft = "@id"; // Request ID command definition
  var view = new Uint8Array(3); // View to contain the command being sent

  // Fill view with the commands individual bytes
  for(var x = 0; x < commandLeft.length; x++)
  {
    view[x] = commandLeft.charCodeAt(x);
  }

  device.send(view.buffer); // Send command
}

device.open({ stopBits: 0, bitRate: 9600, ctsFlowControl: 0 });

‘.open’ is called on serialDevice objects and opens a connection with a device. The default values for all the available fields can be found here:

https://github.com/LLK/scratchx/wiki#serial-device-support

device.set_receive_handler(function(data) { 
  // Insert code here 
});

‘.setreceivehandler’ is called on serialDevice objects and accepts a function as its parameter. After this method runs, whenever data is received from the serial device, the specified function will run to handle the data. The function definition must accept a single parameter, which will contain the raw data (stored in a ArrayBuffer) that the device sends back.

console.log(device.constructor.name);

Use ‘.constructor.name’ on a device object to check that it is indeed a serialDevice object.

Pseudo Multi-Threading

This is a uncommon issue in JavaScript since JavaScript is not often required to run multiple threads at the same time. In addition, JavaScript is, for the most part, a single-threaded language, so an attempt at using multi-threaded processes should not be undertaken lightly as it can become very complicated and very messy.

An example of such a situation where it was required would be the Tracey robot. In order to send a read request for the infra-red sensors and then interpret the 2 bytes that were send back, multiple threads were required. This was because the receive handler in the JavaScript code for the device could not be called, and only ever ran on receival of data, however, the event was added to the queue of functions in its scope (i.e. after the function that sent a read request and interpreted the data), meaning data was either not read at all or was old. Multiple threads fixed this issue by having a separate thread that repeatedly sent the read requests, the data of which was then stored in a buffer object which could then be read at any point.

If you encounter such a situation, the simplest solution could be the use of the JavaScript command setTimout (e.g. setTimeout(function(){ alert("Hello"); }, 3000);). The setTimeout behaves in a similar way to multiple threads, waiting a specified number of milliseconds before running the code. In addition, setInterval also can be used in multiple threads to have a process continuously running in seperate thread (e.g. setInterval(function(){ alert("Hello"); }, 3000);). It works in a similar way to setTimeout but differs in that instead of a delay before running, it runs the code on every interval, specified by the number of milliseconds as its second parameter.

Here is an example which repeatedly calls a function in the background every 120 milliseconds after a 1 second delay:

setTimeout(setInterval(function(){
        if(device)
        {
            sendPinCommand(1);
        }
    }, 120), 1000);
